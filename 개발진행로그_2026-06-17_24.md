# 개발진행로그 24

## 목표

Railway가 정적 HTML이 아니라 Next.js 서버 앱으로 빌드하도록 배포 감지 경로를 더 명확히 한다.

## 근거

- Railway 공식 문서 기준으로 루트 `Dockerfile`이 있으면 Dockerfile을 사용해 서비스를 빌드한다.
- Config as Code의 `railway.json`은 배포 시 대시보드 설정과 병합되며 코드 설정이 우선한다.
- 현재 공개 URL은 여전히 기존 정적 HTML을 서빙하므로, 저장소 차원에서 Dockerfile 빌드를 명시한다.

## 변경 내용

- 루트 `Dockerfile`을 추가했다.
- Docker 빌드 컨텍스트에서 `.git`, `.next`, `node_modules`, `.env` 등을 제외하는 `.dockerignore`를 추가했다.
- `railway.json` 빌더를 `DOCKERFILE`로 명시하고 `dockerfilePath`를 지정했다.
- README 배포 섹션을 Dockerfile 기준으로 갱신했다.

## 검증 결과

- `railway.json` JSON 파싱 확인
- `npm run build` 통과

## 확인 필요

- 현재 로컬 환경에는 `docker` 실행 파일이 없어 Docker 이미지 빌드는 직접 검증하지 못했다.
- Railway 배포 로그에서 Dockerfile 빌드가 사용되는지 확인해야 한다.
- 배포 후 `/api/health`, `/api/version`이 JSON으로 응답해야 한다.

## 푸시 후 공개 URL 확인

- 커밋 `89337d1` 푸시 후 즉시 확인했지만 `/api/version`은 404였다.
- 1분 대기 후 다시 확인해도 `/api/version`은 404였다.
- 루트 응답은 계속 `content-length: 24489`의 기존 정적 HTML이었다.
- 따라서 현재 공개 URL 문제는 저장소 코드 설정보다 Railway 서비스의 연결 브랜치, Root Directory, 배포 대상 서비스, 자동 배포 로그 확인이 필요하다.
