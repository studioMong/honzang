# 개발 진행 로그 - 2026-06-18 21

## 목표
- 같은 자료 유형 안에서도 CSV 헤더 구조가 다른 파일을 여러 개 보관할 수 있게 한다.
- 실제 은행, 카드사, PG사별로 컬럼 구조가 달라도 기존 매핑 템플릿이 덮어써지지 않게 한다.

## 변경 내용
- `src/app/api/imports/route.ts`
  - CSV 매핑 템플릿 저장 기준을 `자료 유형 + 템플릿명`에서 `자료 유형 + 헤더 시그니처`로 변경했다.
  - 같은 헤더 시그니처가 있으면 기존 템플릿을 갱신한다.
  - 같은 자료 유형에 새 헤더 시그니처가 들어오면 새 템플릿을 생성한다.
  - 첫 템플릿은 `기본 템플릿`, 이후 템플릿은 자료 유형별 순번 이름으로 구분한다.
- `scripts/verify-db-workflow.ts`
  - 검증용 CSV 헤더에 고유 마커를 붙여 실제 스테이징/로컬 DB의 기존 템플릿을 덮어쓰지 않게 했다.
  - BANK 자료 유형에 서로 다른 헤더 시그니처 2개를 업로드하고, 두 템플릿이 동시에 보존되는지 확인한다.
  - 검증 중 생성된 템플릿 ID를 추적해 정리 단계에서 `DELETE /api/csv-templates`로 삭제한다.

## 검증
- `npm run typecheck` 통과.
- `npm run lint` 통과.
- `npm run verify:mvp` 통과.

## 확인하지 못한 항목
- `npm run verify:db-workflow`는 실제 Postgres가 연결된 로컬 또는 스테이징 앱 서버가 필요하므로 이번 로컬 샘플 환경에서는 실행하지 않았다.
- 대신 해당 스크립트의 TypeScript 검증은 `npm run typecheck`에 포함되어 통과했다.

## 판단
- 실제 사용자가 국민은행, 토스뱅크, 카드사, PG 정산 파일처럼 같은 자료 유형 안에서 서로 다른 CSV 구조를 쓰더라도 기존 매핑을 잃지 않는다.
- DB 검증 스크립트도 운영 템플릿을 덮어쓰지 않는 방향으로 안전해졌다.
