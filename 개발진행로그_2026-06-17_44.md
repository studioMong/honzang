# 개발 진행 로그 44

## 목표

- 전체 백업을 내려받을 수만 있고 복원할 수 없는 운영 공백을 줄인다.
- Railway Postgres 재생성, 서비스 이전, 장애 복구 상황에서 백업 JSON으로 워크스페이스 데이터를 다시 넣을 수 있게 만든다.

## 작업 내용

- `POST /api/backups/restore` API를 추가했다.
- 백업 JSON이 `혼자장부` `backupVersion: 1` 형식인지 Zod로 검증한다.
- 복원은 Postgres DB 모드에서만 실행한다.
- `dryRun: true` 요청은 DB를 교체하지 않고 백업 형식과 복원 예정 개수만 반환한다.
- API 호출에는 `confirmReplace: true`를 요구한다.
- 복원 방식은 `replace`로 동작한다.
  - 현재 기본 회사의 거래, 증빙, 분개, 리포트, 거래처, 자동 분류 규칙, CSV 템플릿, 업로드 배치 메타데이터를 삭제한다.
  - 백업 JSON의 회사 정보와 워크스페이스 데이터를 다시 생성한다.
- 거래 복원을 위해 `AppTransaction`과 DB 직렬화에 다음 원천 필드를 추가했다.
  - `importBatchId`
  - `sourceRowNumber`
  - `approvalNumber`
- 복원 시 기존 백업 거래 ID를 새 DB 거래 ID로 매핑해 증빙, 분개, 검토 항목 연결을 유지한다.
- 설정 화면 `전체 백업` 섹션에 `백업 복원` 버튼을 추가했다.
- 복원 전 백업 JSON dry-run 검사를 실행하고, 복원 예정 거래/증빙/분개 개수를 확인창에 표시한다.
- 복원 성공 후 전체 데이터를 다시 불러와 화면 상태를 DB와 맞춘다.
- README의 MVP 범위에 백업 JSON 복원을 추가했다.

## 복원 범위

- 회사 기본정보
- 계정과목
- CSV 템플릿
- 업로드 배치 메타데이터
- 거래
- 증빙 메타데이터와 DB 보관 소형 파일 데이터
- 분개와 분개 라인
- 저장 리포트
- 거래처 기본값
- 자동 분류 규칙
- 검토 항목 상태

## 제한

- 이번 복원은 JSON 백업 기준이다.
- 당시에는 ZIP 백업 안의 원본 CSV 파일을 자동으로 다시 DB `originalFileText`에 넣는 기능은 아직 추가하지 않았다.
  - 이후 [[개발진행로그_2026-06-17_46]]에서 백업 JSON의 `originalImportFiles` 기반 원본 CSV 복원을 추가했다.
- 복원은 현재 DB 데이터를 교체하는 작업이므로 운영 환경에서는 복원 전 별도 백업을 먼저 내려받아야 한다.
