# 개발 진행 로그 34

## 목표

- 소형 증빙 파일을 앱 안에서 바로 보관하고 다시 내려받을 수 있게 만든다.
- 신고 준비 패키지 JSON에 증빙 목록을 포함해 회계사 또는 신고 검토자가 거래와 증빙을 같이 볼 수 있게 한다.

## 작업 내용

- 증빙 등록 폼에서 750KB 이하 파일을 Data URL로 읽어 DB에 저장하도록 추가했다.
- 증빙 파일 본문, MIME 타입, 파일 크기는 `Evidence.rawPayload` JSON 필드에 보관하도록 했다.
- 증빙 목록에서 DB에 보관된 파일은 `다운로드` 버튼으로 다시 받을 수 있게 했다.
- 신고 준비 패키지 JSON의 `tables.evidences`에 증빙 발행일, 유형, 거래처, 매칭 거래, 금액, 파일 보관 상태, 파일 크기를 포함했다.
- README의 MVP 범위에 소형 증빙 파일 DB 보관 및 다운로드를 추가했다.

## 검증

- `npm run typecheck` 통과
- `npm run lint` 통과
- `npm run verify:samples` 통과
- `npm run build` 통과
- `npm run smoke:prod` 통과
  - 로컬 standalone 서버 기준 `http://127.0.0.1:3100`
- 로컬 개발 서버 API 확인
  - `POST /api/evidences`에 파일 Data URL, MIME 타입, 크기 포함 요청 성공
- 로컬 증빙 화면 SSR 응답 확인
  - `증빙 목록` 문구 확인
  - `750KB 이하 파일은 DB에 보관합니다` 문구 확인

## 공개 배포 상태

- `https://honzang-production.up.railway.app/`는 200 응답
- `https://honzang-production.up.railway.app/api/version`은 404 응답
- `https://honzang-production.up.railway.app/api/health`는 404 응답
- 로컬 빌드 산출물에는 `/api/version`, `/api/health`가 포함되므로, Railway 공개 도메인이 최신 Next 서버가 아니라 이전 정적 배포 또는 다른 서비스에 연결된 상태로 판단한다.

## 비고

- 브라우저 headless 검증은 로컬 Playwright 브라우저 실행 파일이 없어 실행하지 못했다.
- 현재 방식은 MVP용 소형 파일 보관이다. 큰 PDF, 이미지 원본, 장기 보관 증빙은 추후 S3/R2 같은 오브젝트 스토리지로 분리하는 편이 낫다.
