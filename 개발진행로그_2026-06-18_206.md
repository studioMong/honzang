# 개발 진행 로그 206

## 작업 주제
- Railway Postgres에 직접 보관하는 원본 CSV와 증빙 파일의 선택적 암호화 저장 지원.

## 변경 사항
- `src/lib/server/file-encryption.ts`를 추가했다.
  - `HONZANG_FILE_ENCRYPTION_KEY`가 설정되면 AES-256-GCM으로 stored text를 암호화한다.
  - 암호화된 값은 `honzangenc:v1:` prefix로 구분한다.
  - 키가 없으면 기존 로컬/샘플 흐름처럼 평문 저장을 유지하고, 기존 평문 데이터도 그대로 읽는다.
  - 잘못된 키나 깨진 암호문은 평문을 반환하지 않고 `null`로 처리한다.
- 원본 CSV 저장/조회 경로에 암호화/복호화를 적용했다.
  - `src/app/api/imports/route.ts`
  - 신규 업로드와 중복 배치 원본 보완 시 `originalFileText`를 암호화 저장한다.
  - 원본 CSV 다운로드 API는 복호화된 원문만 반환하고, 키가 없어서 복호화할 수 없으면 409를 반환한다.
- 증빙 파일 저장/조회 경로에 암호화/복호화를 적용했다.
  - `src/app/api/evidences/route.ts`
  - `src/lib/server/serializers.ts`
  - DB에 보관하는 `rawPayload.fileDataUrl`은 암호화 저장하고, API 응답/백업 export 전에는 복호화한다.
- 백업 복원 경로에 암호화 저장을 적용했다.
  - `src/app/api/backups/restore/route.ts`
  - 백업 JSON으로 복원되는 원본 CSV와 증빙 fileDataUrl도 현재 환경의 파일 암호화 키 기준으로 저장한다.
- 운영 준비 점검에 `파일 암호화 키` 항목을 추가했다.
  - 프로덕션에서 `HONZANG_FILE_ENCRYPTION_KEY`가 없으면 필수 누락으로 표시한다.
- `.env.example`과 README에 `HONZANG_FILE_ENCRYPTION_KEY`를 추가했다.
- `scripts/verify-file-encryption.ts`를 추가했다.
  - 키 없음 평문 호환, 키 있음 암호화 round-trip, 잘못된 키 방어, 중복 암호화 방지, 코드 연결 지점을 검증한다.
- `verify:mvp`에 `verify:file-encryption` 단계를 추가했다.

## 비고
- 이번 변경은 Postgres 직접 파일 보관 구조에서 운영 보안 리스크를 줄이는 작업이다.
- 기존 평문 데이터는 읽기 호환을 유지한다. 운영에서 키를 설정한 이후 신규 원본 CSV와 증빙 파일부터 암호화 저장된다.
- Railway 공개 URL의 legacy/static 연결 문제는 이번 코드 변경과 별개로 계속 남아 있다.
