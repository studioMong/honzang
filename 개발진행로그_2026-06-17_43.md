# 개발 진행 로그 43

## 목표

- Railway 공개 도메인이 최신 Next.js 앱을 서빙하지 않는 문제를 진단 가능한 상태로 만든다.

## 확인 결과

- Railway 공개 URL 검증 당시 기능 커밋은 `73ed00d Add workspace backup export`까지 푸시되어 있었다.
- 이후 진단 개선 커밋 `2fc1a5d Improve Railway deployment diagnostics`까지 원격 `main`에 푸시했다.
- `https://honzang-production.up.railway.app/api/version`은 현재 HTTP 404를 반환한다.
- 저장소에는 `src/app/api/version/route.ts`, `src/app/api/health/route.ts`가 존재한다.
- 로컬 production standalone 빌드에서는 API와 화면 검증이 통과했다.
- 따라서 현재 문제는 앱 코드에서 API 라우트가 빠진 것이 아니라 Railway 공개 도메인이 최신 Next.js 서비스에 연결되어 있지 않거나, 최신 배포가 실패하고 이전 배포가 유지되는 상황으로 보는 것이 맞다.

## 작업 내용

- `npm run verify:railway` 실패 시 추가 진단을 출력하도록 개선했다.
- 실패 시 다음 경로의 HTTP 상태, content-type, 응답 크기를 함께 출력한다.
  - `/`
  - `/api/version`
  - `/api/health`
  - `/manifest.webmanifest`
- 루트 응답에 `프로젝트 정리` 문구가 있으면 레거시 정적 프로젝트 요약 페이지로 보인다는 진단을 출력한다.
- API 경로가 JSON이 아닌 응답을 반환하면 짧은 body preview를 출력한다.

## Railway 대시보드 점검 항목

- `honzang-production.up.railway.app` 도메인이 Next.js 앱 서비스에 붙어 있는지 확인한다.
- 배포 소스가 `github.com/studioMong/honzang`, 브랜치가 `main`인지 확인한다.
- 배포 로그에서 Dockerfile 빌더가 사용되는지 확인한다.
- 배포 로그에서 `npm run build`와 `npm run db:deploy`가 성공했는지 확인한다.
- 서비스 환경 변수에 `DATABASE_URL`이 Railway Postgres 연결 문자열로 들어가 있는지 확인한다.
- 최신 배포 커밋이 GitHub `main` 최신 커밋과 일치하는지 확인한다.
- 배포가 실패했다면 실패 로그를 기준으로 DB 마이그레이션, 빌드, 시작 명령 중 어디에서 멈췄는지 확인한다.

## 검증

- `npm run lint` 통과
- `npm run verify:railway` 실패 확인
  - `/` HTTP 200, 레거시 정적 프로젝트 요약 페이지로 진단됨
  - `/api/version` HTTP 404
  - `/api/health` HTTP 404
  - `/manifest.webmanifest` HTTP 404
- 실패 자체는 현재 공개 도메인 연결 또는 Railway 배포 상태 문제를 확인하기 위한 기대 결과다.
